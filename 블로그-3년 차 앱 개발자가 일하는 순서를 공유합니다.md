# 3년 차 앱 개발자가 일하는 순서를 공유합니다

## 들어가며

3년 차 개발자로 일하면서 내 방식을 정리해볼 필요가 있다고 생각했다. 완벽하지 않고 때로는 부족하지만, 현실적인 내 개발 과정을 솔직하게 공유해보려고 한다.

## 1. 기획서 검토에 대한 솔직한 고백

**기획서를 대략적으로 확인한 상태라 대략적인 화면만 확인했습니다.**

퇴근 후에 진행하는 프로젝트이다보니 기획서를 꼼꼼히 읽기보다는 UI 목업이나 와이어프레임 위주로 빠르게 훑어보는 경우가 많다.

하지만 이렇게 하면 나중에 문제가 생긴다:
- 개발 중간에 "어? 이 경우는 어떻게 처리하지?" 같은 의문이 생김
- 비즈니스 로직을 제대로 이해하지 못해서 다시 기획자에게 물어봐야 함
- 예외 상황이나 엣지 케이스를 놓치게 됨

**개선하려는 방향**:
- 가지고 있는 리소스가 부족하다 보니 기획자 혹은 혼자 우선 순위가 높은 기능만 확인을 한다.
- 놓칠 수 있는 부분이 있지만 시간이 부족하다.

## 2. 소통의 필요성을 느끼다

**결국 리뷰한 내용을 기획자에게 발표나 대화가 필요할 것 같다.**

개발하다 보면 혼자 고민으로 끝나는 게 아니라, 결국 기획자나 팀원들과 얘기해야 할 상황이 생긴다. 그럼 처음부터 소통하는 게 낫지 않을까?

내가 경험한 소통의 효과:
- 잘못 이해했던 부분을 일찍 발견할 수 있음
- 기획의 의도를 이해할 확률이 높아짐
- 기술적 제약사항을 미리 공유해서 기획 수정이 가능함
- 기획자가 놓친 부분이나 추가 요구사항을 발견할 수 있음

**실제 적용**:
기획서 리뷰 후에는 30분 정도 기획자와 브리핑 시간을 갖자. 내가 이해한 내용을 설명하고 궁금한 점들을 미리 해결하자.

## 3. 와이어프레임의 한계를 깨닫다

**현재는 와이어프레임을 작성하고 있는 상태라 동작 흐름에 대해서 작성할 필요가 없다고 판다.**
**화면 이동 이벤트 및 전달할 데이터에 대해서 고민하지 못했다.**

처음에는 와이어프레임만 보고 개발하면 된다고 생각했다. 하지만 실제로는 정적인 화면 구성보다 동적인 부분이 더 복잡하고 중요하다는 걸 깨달았다.

내가 놓쳤던 것들:
- 사용자가 어떤 순서로 화면을 이동하는지
- 화면 간에 어떤 데이터를 주고받는지
- 뒤로가기나 앱 재실행 시 상태를 어떻게 처리할지
- 네트워크 오류나 로딩 상태는 어떻게 보여줄지

**지금 하는 방법**:
와이어프레임를 복사해서 프론트 개발에 필요한 데이터나 상태를 메모한다.

## 4. 백엔드와의 협업 필요성

**추가적으로 화면에서 어떤 데이터가 필요하다는 것에 대해서 백엔드와 소통하기 필요하다.**

프론트엔드 개발하다가 "아, 이 데이터는 어디서 가져오지?" 하고 나중에 깨닫는 경우가 너무 많았다. API 명세서가 있어도 실제 화면을 만들다 보면 부족한 부분들이 보인다.

**개선 방법**:
화면 설계가 끝나면 백엔드 개발자와 API 리뷰 시간을 갖자. 각 화면별로 필요한 데이터를 체크리스트로 만들어서 함께 검토하자.

## 5. 나의 작업에 대한 의문

**내가 작성한 내용이 실제로 필요한지 다시 생각해보자**

모든 걸 완벽하게 하려고 하다 보면 오히려 비효율적일 수 있다. 코드든, 기능이든, 문서든 말이다. 특히 시간과 리소스가 한정된 상황에서는 더욱 그렇다.

내가 하는 자기 검열:
- 이 기능이 정말 사용자에게 필요한가?
- 지금 구현해야 할 우선순위가 맞나?
- 과도하게 복잡하게 만들고 있는 건 아닌가?
- 나중에 유지보수할 사람 입장에서는 어떨까?

가끔은 "일단 동작하는 버전부터 만들자"는 마음가짐도 필요하다고 생각한다.

## 6. 작은 팀에서의 개발

**6번 이후로는 밑에 출시전 제품이고 처음 개발하는 밑에 사람 1명이라 큰고민이 필요 하지 않을 것 같다.**

처음에는 이렇게 생각했다. 작은 팀이니까, 출시 전 제품이니까 그렇게 복잡하게 생각할 필요 없다고.

하지만 실제로는:
- 작은 팀일수록 한 사람의 실수가 큰 영향을 미침
- 코드 리뷰나 지식 공유가 제한적이라 더 신중해야 함
- 초기 아키텍처나 코딩 스타일이 나중에 큰 기술 부채가 될 수 있음
- 새로 합류하는 팀원을 위해 문서화나 코드 가독성이 더 중요함

**현재 생각**:
작은 팀이라고 해서 대충 할 게 아니라, 오히려 더 기본에 충실하게 해야 한다. 복잡한 패턴보다는 단순하고 명확한 코드, 충분한 주석과 문서화가 더 중요하다.

## 마무리
이 글을 쓰면서 내가 어떤 부분에서 부족한지, 어떤 방향으로 개선해야 할지 정리할 수 있어서 도움이 되었다. 앞으로도 계속 시행착오를 겪으면서 더 나은 방식을 찾아가려고 한다.

다른 개발자들도 비슷한 고민을 하고 있을 것이다. 완벽하지 않아도 서로의 경험을 공유하면서 함께 성장해나갔으면 좋겠다.
